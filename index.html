Syntax

Clarence's syntax is mainly inspired by Clojure. This allowes us to use a pretty sweet and readable synax. So don't worry Clojure devs... We provide a clojure similar vm that is even much more optimized than the older one. In Clarence we use dynamic bytecode compilation for a speedy solution. As a round up, we've got some more tricks on board... Clarence uses a such high perfomable syntax, that can even write code for you! It uses lightweight macros for it's concurrency. 

;   A little web-server running on port: 3000

(def http (js/require "http"))
(def server (. http (createServer
  (fn [req res]
    (. res (end "Hello, world"))))))
    (. server (listen 3000))
    (. js/console (log "Listening port 3000"))

General specifications

Clarence's type system is built in using different compile channels (think crystals ones) that interact parallel while compilation with the error checker. This enables us to detect stinky bugs quite before execution and not while runtime startup.

As we already said before, we use lightweight macros to solfe daily problems in multithreading. Our concurrency system is quite a powerful one. It was built using itselfes macros :) As in crystal or in clojure, clarence supports the definition of a channeling system. This is a quit good one, because we are now able to interact directly with other macros through channels. They are kinda secure sleeve around the macros. In fact, metaprogramming is pendent on clarence's macro system. That's why we feature such a strong multithreading system.

However, we don't need a such gigantic system when we'd have a low error handling. This was one of my problems while constructing the Clarence language. Finally tt's error handling supports direct and smooth analization of macro funcions wich is quit important in a multithreading system. Though it features a precise error reference before execution.

Macros

We've talked a lot of macros... But what the heck are macros?? I'll explain it. Macros are our answer to metaprogramming in clarence. They are similar to csharps threads, or to Darts fiber system. Macros are quite helpful in situations in which you wan't to execute untrused code on a virtual machine. They are kinda a tooling method to execute functions parallel to compilation. Macros interact with each other using a channeling system. They work as a fiber-log system. Think Crystals channels or see the example. 

; A little macro call

(ns clr.macro)                                  ; namespace definition
(def defmacro (fn [var1 & var2]                 ; declaring a macro with 2 values (var1, var2)
  (do (def ~var1 (fn ~@var2)                    ; exchanging the values of both variables
  (do (def ~var2 (fn ~@var1))                   ; exchanging the values of both variables
    (set! (. ~var1 macro) true))                ; setting var1 to value true
    (set! (. ~var2 macro) nil)))                ; setting var 2 to value nil
    (set! (. defmacro macro) true)              ; setting the macro itselfe to value true

(defmacro defn [var1 & var2]                    ; calling the macro funcion
  (def ~var2 (fn ~var2)))                       ; exchanging the variables
  (defn var2 "Variable 2 has this value!")      ; define the value for variable 2
    (println var1 .. var2))                     ; printing the variable 2
